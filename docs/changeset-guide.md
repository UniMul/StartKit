# Changeset Guide

This repository uses Changesets for release notes and changelog generation.

## Purpose

A changeset file is a small markdown file under:

```text
.changeset/
```

It describes a user-facing change and the release bump level.

Examples:
- patch
- minor
- major

---

## Basic flow

### 1. During a pull request
Add a changeset file for user-facing changes.

Create it with:

```bash
npx changeset
```

or create a file manually in `.changeset/`.

### 2. Before release
Run:

```bash
npx changeset version
```

This updates:
- `CHANGELOG.md`
- `VERSION` or other version metadata

### 3. Commit the generated release changes
Commit the updated files:
- `CHANGELOG.md`
- `VERSION`
- any changes generated by Changesets

### 4. Tag and release
Create a tag and publish the release.

---

## Recommended PR rule

If a pull request changes user-facing behavior, include a `.changeset/*.md` file.

Docs-only changes may be excluded depending on project policy.

---

## Manual example

```md
---
"start-kit": minor
---

Add dependency confirmation UX for module installs.
```

---

## Relationship to CHANGELOG.md

- `.changeset/*.md` = release note fragments
- `CHANGELOG.md` = generated public changelog

Users should read `CHANGELOG.md`.
Contributors should write `.changeset/*.md`.
